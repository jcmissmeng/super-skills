# Project Onboard — 项目系统化理解与文档持续维护

## 概述

三阶段九步骤流程，从零开始系统化理解已有项目，逆向产出 spec.md / design.md / tasks.md，并在后续开发中持续同步文档。

## SDD文档目录

所有SDD文档存放在项目根目录的 `docs/spec/` 下（如不存在则自动创建）：

```
项目根目录/
└── docs/
    └── spec/
        ├── spec.md      # 功能规格（做什么）
        ├── design.md    # 架构设计（怎么做）
        └── tasks.md     # 任务清单（还差什么）
```

## 核心流程

### 阶段一：探索理解

#### 步骤1：鸟瞰全局

```
请帮我 zoom-out，从高层视角理解这个项目的整体架构、
模块划分、核心依赖关系和业务流程。
```

#### 步骤2：深度探索

```
请深度探索这个项目，我需要了解：
1. 目录结构及各模块职责
2. 核心模块间的调用关系和数据流
3. 配置/环境变量的体系及消费链路
4. 入口文件的启动编排逻辑
5. 对外接口（API/CLI/事件）的完整清单
6. 第三方依赖及其用途
```

#### 步骤3：提炼领域语言

```
请提取这个项目的领域术语表（ubiquitous-language），
覆盖：业务概念、技术概念、模块概念，
消除命名歧义，建立统一术语。
```

### 阶段二：结构化文档化（SDD逆向补全）

#### 步骤4：初始化SDD目录

```
在项目根目录下创建 docs/spec/ 目录（如不存在），
用于存放 spec.md / design.md / tasks.md。
```

#### 步骤5：逆向写 spec.md

```
请基于现有代码逆向梳理 spec.md，覆盖：
1. 功能清单：列出所有已实现功能，区分核心功能和辅助功能
2. 业务规则：每个功能的关键业务逻辑、校验规则、状态流转
3. 用户场景：主要使用链路和操作流程
4. 约束条件：平台限制、权限要求、版本依赖、已知限制
5. 配置项说明：所有配置字段及其含义、默认值、影响范围
6. 接口契约：对外暴露的API/CLI/事件的输入输出规范
```

#### 步骤6：逆向写 design.md

```
请基于现有代码逆向梳理 design.md，覆盖：
1. 架构模式：整体分层策略和模块划分原则
2. 目录结构：每个目录/模块的职责和对外接口
3. 技术选型：关键依赖的选择原因和替代方案考量
4. 数据流：核心数据从输入到输出的完整流转路径
5. 关键设计决策：已做出的架构选择及其trade-off
6. 部署方式：部署架构、环境要求、启动方式
7. 扩展点：预留的扩展机制（插件/钩子/配置化）
```

#### 步骤7：写 tasks.md

```
请基于 spec.md 和 design.md 梳理 tasks.md：
1. 已完成的功能（标记done）
2. 半成品/待完善的功能（标记in_progress）
3. 待开发功能（标记pending，来源：TODO/Issue/README计划）
4. 按优先级排列，每个任务给出验收标准
```

### 阶段三：架构优化

#### 步骤8：架构改进

```
请分析这个项目的架构改进机会：
1. 模块间耦合度和依赖方向是否合理
2. 是否有重复逻辑可抽取
3. 配置体系是否足够灵活
4. 错误处理和容错机制是否完善
5. 测试覆盖情况
6. 性能瓶颈或安全隐患
```

#### 步骤9：验证理解

```
请用 grill-with-docs 验证 spec.md 和 design.md 的准确性：
对照实际代码，质疑文档中可能的错误或遗漏，
确保文档与代码实现一致。
```

## 执行策略

- 步骤1-3 可一次性并行发出
- 步骤4 必须先执行（创建docs/spec/目录），步骤5-7 依赖该目录
- 步骤5-6 可并行，spec和design互不依赖
- 步骤7 依赖5和6的产出
- 步骤8-9 依赖阶段二的产出，需串行

## 开发中持续更新文档
