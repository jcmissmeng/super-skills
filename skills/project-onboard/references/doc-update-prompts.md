# 文档持续更新提示词模板

## 速查口诀

| 场景 | 更新文档 | 原则 |
|------|---------|------|
| 加功能 | spec + design + tasks + README | 先更新再实现 |
| 改Bug | spec(规则) + design(容错) | 仅当暴露规格遗漏时 |
| 重构 | design(必) + spec(仅行为变) | 架构变必更新design |
| 改配置 | design(部署) + spec(约束) + README | 用户可见必更新README |
| 发版 | 全量对齐检查 | 文档即合约 |

---

## 功能开发前

```
我要开发 [功能描述]，请先检查 spec.md 和 design.md 中
相关内容，确认是否需要更新规格或设计文档后再开始实现。
```

## 功能开发后

```
[功能]已实现，请更新以下文档：
1. spec.md - 补充新功能的功能说明、业务规则、用户场景
2. design.md - 补充新模块/接口/数据流的设计说明
3. tasks.md - 将对应任务标记为done，补充新发现的待办
4. README.md - 更新功能列表、使用说明（如有用户可见变化）
```

## Bug修复后

```
[Bug描述]已修复，请检查是否需要更新：
1. spec.md - 业务规则/约束条件是否有遗漏
2. design.md - 错误处理/容错机制是否需要补充说明
3. tasks.md - 是否有相关待办需要更新
```

## 重构后

```
[重构内容]已完成，请更新：
1. design.md - 架构/模块关系/数据流的变化
2. spec.md - 仅当对外行为有变化时更新
3. tasks.md - 标记完成，补充重构衍生的待办
```

## 配置/部署变更后

```
[变更内容]已完成，请更新：
1. design.md - 部署方式/环境要求
2. spec.md - 约束条件/配置项说明
3. README.md - 安装/部署/使用说明
```

## 版本发布前

```
即将发布版本，请全面检查文档一致性：
1. spec.md ↔ 实际功能是否对齐
2. design.md ↔ 实际架构是否对齐
3. tasks.md ↔ 实际进度是否对齐
4. README.md ↔ 用户可见信息是否准确
5. CHANGELOG（如有）是否更新
```

## 新接手项目时（首次文档补全）

```
请按 project-onboard 技能的三阶段九步骤流程，
系统化梳理当前项目，产出 docs/spec/ 下的 spec.md / design.md / tasks.md。
```

## 定期文档审计

```
请审计当前项目的文档一致性：
1. spec.md 中的功能是否都有对应代码实现
2. design.md 中的架构描述是否与实际代码结构一致
3. tasks.md 中的任务状态是否与实际进度匹配
4. README.md 是否覆盖所有用户可见变化
5. 列出所有不一致项并给出修正建议
```
